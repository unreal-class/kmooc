## 실습 3주차-1. 회전 컴포넌트를 이용한 회전 기능의 구현


이제 골드 트로피 블루프린트 에디터를 열어 부품을 넣어보는 실습을 진행하겠습니다.

에디터 창의 왼쪽 위를 보면 '컴포넌트(Components)' 패널이 보입니다.
현재는 `StaticMeshComponent`라는 부품이 하나 들어있습니다.
이 부품의 역할은 3D 모델 데이터를 화면에 보여주는 것입니다.
이 부품이 들어있기 때문에 액터를 레벨에 배치했을 때 트로피 모델이 눈에 보이는 것입니다.
컴포넌트 패널 위의 '+추가(Add)' 버튼을 누릅니다.
추가할 수 있는 부품 리스트가 쭉 나오면 회전 기능을 가진 부품을 검색합니다.
`RotatingMovement` 컴포넌트를 찾아서 추가합니다.
이제 컴포넌트 패널에 부품이 2개가 된 것을 확인할 수 있습니다.
컴파일을 하고 세이브를 한 뒤 레벨에서 플레이 버튼을 눌러봅니다.
트로피가 자동으로 회전하는 모습을 볼 수 있습니다.
직접 회전 코딩을 할 필요 없이 부품만 끼워 넣으면 해결됩니다.

부품을 가져온 뒤에는 성능이나 기능을 원하는 대로 튜닝할 수 있습니다. 
자동차 엔진을 사서 장착한 뒤 속도를 높이도록 세팅하는 것과 같습니다. 
부품이 속도 조절 기능을 제공하므로 속성을 쉽게 바꿀 수 있습니다. 
컴포넌트를 선택한 뒤 디테일 패널에서 수정하면 됩니다.

컴포넌트 패널에서 `RotatingMovement`를 선택합니다.
오른쪽 디테일 패널 중간을 보면 `Rotation Rate` 속성이 보입니다.
현재 X축 0, Y축 0, Z축(수직축) 중심으로 초당 180도 회전하도록 설정되어 있습니다.
한 바퀴 도는 데 2초가 걸리는 속도입니다.
더 빠르게 돌리기 위해 Z축 값을 1800으로 변경해 봅니다.
속도를 10배 올렸으므로 1초에 대략 대여섯 바퀴씩 아주 빠르게 돌게 됩니다.
컴파일 후 플레이를 눌러 회전 속도가 빨라진 것을 확인합니다.

언리얼 엔진을 배운다는 것은 이처럼 제공되는 컴포넌트들을 충분히 이해하고 활용하는 것을 뜻합니다. 
추가 버튼을 눌렀을 때 나오는 수많은 컴포넌트를 이해해야 엔진을 제대로 쓸 수 있습니다. 
클래스와 컴포넌트를 활용하는 설계 메커니즘이 조금씩 이해되실 것입니다. 
언리얼 엔진은 AI, 오디오, 카메라 등 수많은 컴포넌트를 기능 단위로 제공하므로 이를 잘 다루어야 합니다.

## 실습 3주차-2. 사용자 컴포넌트 제작

이번에는 두 번째 실습으로 의자를 위아래로 움직이는 상하 운동을 구현해 보겠습니다. 
지난 시간에는 트로피를 움직였지만 이번에는 의자입니다. 어떻게 하면 될까요? 
의자를 선택해 블루프린트 클래스를 만들고, 틱 이벤트에 똑같은 이동 로직을 구현하면 될 것입니다.

하지만 매번 물체마다 이 작업을 반복해야 할까요? 
당연히 아닙니다. 
단순하게 생각하면 기존 코드를 복사해서 붙여넣으면 되겠지만, 만약 소파나 테이블 등 수많은 다른 물체들도 똑같이 상하 운동을 시켜야 한다면 매번 클래스를 만드는 방식은 한계가 있습니다.

의자를 상하 운동시키는 기능에 대해 생각해 봅시다. 
회전 기능처럼 상하 운동 기능도 게임 내에서 아주 많이 쓰입니다. 
아쉽게도 언리얼 엔진이 상하 운동 컴포넌트를 기본으로 제공하지는 않습니다. 
하지만 우리가 직접 이 기능을 가진 부품을 만들어서 필요한 물체마다 끼워 넣는다면 아주 편리할 것입니다. 
그래서 오늘 실습은 우리가 직접 상하 운동 부품을 제작하고 적용해 보는 것입니다. 
컴포넌트 기반 설계에서 아주 중요한 개념이니 잘 따라와 주시기 바랍니다.

콘텐츠 브라우저 빈 곳에서 마우스 우클릭을 합니다.
'블루프린트 클래스'를 클릭합니다.
부모 클래스 선택 창 중간에서 `ActorComponent`를 선택합니다.
액터 컴포넌트는 재사용이 가능하며 다른 액터에 부품처럼 끼워 넣을 수 있는 클래스입니다.
생성된 컴포넌트의 이름을 `FloatingComponent`라고 지정합니다.
생성된 `FloatingComponent`를 더블 클릭하여 에디터를 엽니다.

일반 블루프린트 클래스인 `BP_GoldTrophy`와 이 `FloatingComponent`를 비교해 보겠습니다. 
언뜻 보면 둘 다 이벤트 그래프가 있어서 비슷해 보이지만 중대한 차이점이 있습니다. 
`BP_GoldTrophy`에는 왼쪽 위에 컴포넌트 패널이 있지만, `FloatingComponent`에는 컴포넌트 패널이 없습니다. 
왜냐하면 이 파일 자체가 부품이기 때문입니다. 부품 안에 또 다른 부품이 존재할 수는 없습니다.

또 하나의 큰 차이점은 뷰포트(Viewport) 탭이 없다는 점입니다. 
트로피 블루프린트는 스태틱 메쉬라는 외형 부품을 가지고 있어서 모양을 시각적으로 보여주는 뷰포트가 필요합니다. 
반면 우리가 만드는 이 부품은 위아래로 움직이는 기능만 담당할 뿐 시각적인 외형이 없기 때문에 뷰포트가 존재하지 않는 것입니다.

컴포넌트 에디터의 이벤트 그래프를 보면 익숙한 `BeginPlay`와 `Tick` 이벤트가 있습니다.
지난 시간 골드 트로피에 작성했던 상하 운동 로직을 그대로 활용하겠습니다.
`BP_GoldTrophy` 이벤트 그래프로 이동하여 상하 운동 관련 노드들을 드래그하여 선택합니다.
`Ctrl + C`로 노드들을 복사합니다.
다시 `FloatingComponent` 이벤트 그래프로 돌아와 `Ctrl + V`로 붙여넣고 각 핀을 연결합니다.
이 상태로 컴파일 버튼을 누르면 변수가 선언되지 않았다는 에러 메시지가 많이 발생합니다.
로직 노드만 복사되고 변수 선언은 복사되지 않았기 때문입니다.
에러가 난 `StartLocation` 변수 노드에서 마우스 우클릭을 합니다.
팝업 메뉴에서 '변수 생성(Create Variable)'을 클릭하여 자동으로 변수를 등록합니다.
`Frequency`와 `Amplitude` 변수 노드도 동일하게 우클릭하여 자동으로 변수를 생성해 줍니다.
컴파일을 다시 진행하여 변수 관련 에러가 사라진 것을 확인합니다.

하지만 여전히 `GetActorLocation`과 `SetActorLocation` 부분에 에러가 남아있을 것입니다. 
이 함수들은 액터의 위치를 다루는 함수인데, 현재 우리가 작업 중인 클래스는 액터가 아니라 컴포넌트(부품)이기 때문입니다. 
부품 자체는 스스로 위치를 가질 수 없고, 자신을 소유하고 있는 본체 제품의 위치를 기준으로 작동해야 합니다. 
부품은 홀로 존재할 수 없으며 항상 액터 안에 포함되어야 의미가 있습니다. 
따라서 이 부품을 담고 있는 소유주 액터, 즉 '오너(Owner)'를 찾아 연결해 주어야 합니다.

`GetActorLocation` 노드의 타겟 핀을 드래그하여 빈 곳에 놓은 뒤 `GetOwner`를 검색합니다.
컴포넌트 전용 `GetOwner` 함수를 선택하여 소유주 액터를 지정합니다.
이렇게 하면 부품을 소유한 오너 액터의 위치 값을 정상적으로 가져올 수 있게 됩니다.
뒷부분의 `SetActorLocation` 노드의 타겟 핀도 동일하게 드래그하여 `GetOwner` 노드와 연결해 줍니다.
다시 컴파일 버튼을 누르면 모든 에러가 사라지고 정상적으로 컴파일이 완료됩니다.
왼쪽 변수 패널에서 `Frequency`(진동수)를 선택하고 초기값을 2로 설정합니다.
`Amplitude`(진폭) 변수를 선택하고 초기값을 50으로 설정한 뒤 컴파일하고 저장합니다.

부품 생성이 완료되면 아이콘이 조립식 퍼즐 모양으로 변경된 것을 볼 수 있습니다. 
이제 이 부품을 액터에 장착하여 테스트해 보겠습니다. 
우리의 목표는 의자를 위아래로 출렁거리게 만드는 것입니다.

스타터 콘텐츠의 프롭스 폴더에서 의자(`SM_Chair`) 에셋을 레벨에 배치합니다.
의자를 공중에 살짝 띄운 뒤 디테일 패널에서 모빌리티 속성을 '무브블(Movable)'로 변경합니다.
의자가 선택된 상태에서 오른쪽 디테일 패널의 컴포넌트 영역을 확인합니다.
콘텐츠 브라우저에 있는 `FloatingComponent`를 드래그하여 의자의 컴포넌트 영역에 떨어뜨려 추가합니다.
플레이 버튼을 누르면 의자가 위아래로 움직이는 것을 확인할 수 있습니다.
같은 방식으로 원형 테이블도 레벨에 배치하고 모빌리티를 '무브블'로 변경합니다.
테이블에도 동일하게 `FloatingComponent`를 드래그하여 추가합니다.
플레이를 누르면 테이블 역시 위아래로 잘 움직이는 것을 볼 수 있습니다.

이것이 바로 컴포넌트 기반 설계의 강력함입니다. 
부품을 한 번 잘 만들어두면 어떤 액터에든 장착하는 즉시 해당 기능이 구현됩니다. 
이 기법은 언리얼 엔진뿐만 아니라 객체지향 프로그래밍이나 디자인 패턴에서 '컴포넌트 기반 클래스 설계' 또는 '합성을 이용한 설계'라고 부르는 매우 중요한 기술입니다. 
여러분이 C++로 게임을 개발할 때도 이러한 구조로 코딩할 수 있는 능력을 갖추어야 합니다.

만약 상하 운동을 하는 의자나 벤치를 10개 넘게 배치해야 한다면, 일일이 액터를 배치하고 부품을 10번 끼워 넣는 방식은 비효율적입니다. 
더 좋은 방법은 애당초 상하 운동 기능이 포함된 의자 클래스를 새로 정의해서 가져다 쓰는 것입니다.

콘텐츠 브라우저에서 소파(`SM_Couch`) 에셋을 레벨에 배치하고 모빌리티를 '무브블'로 변경합니다.
상단의 블루프린트 생성 버튼을 눌러 새 클래스를 만듭니다.
클래스 이름을 `BP_FloatingCouch`라고 지정하고 저장합니다.
열린 블루프린트 에디터 창의 왼쪽 위 '+추가' 버튼을 누릅니다.
우리가 만든 `Floating` 컴포넌트를 검색하여 추가합니다. (커스텀 항목으로 분류되어 있습니다.).
컴파일과 저장을 진행합니다.
이제 레벨에 `BP_FloatingCouch` 블루프린트 액터를 여러 개 드래그하여 배치합니다.
플레이를 누르면 배치된 소파들이 일제히 위아래로 출렁거리며 움직이는 모습을 확인할 수 있습니다.

컴포넌트를 잘 만들어두면 재사용성이 극대화되어 개발 속도가 비약적으로 빨라집니다. 
전체적인 흐름과 컴포넌트의 유용성이 잘 이해되셨을 것으로 생각합니다.
오늘 수업은 마치겠습니다.

## 플로팅 컴포넌트 최적화

실습에 앞서 지난 시간에 만든 플로팅 컴포넌트를 먼저 수정하겠습니다. 
현재 코드는 기능상으로는 전혀 문제가 없지만, 최적화 관점에서 큰 낭비를 안고 있습니다. 
바로 이벤트 틱에서 1초에 60번씩 'Get Owner' 노드를 반복 호출한다는 점입니다. 블루프린트에서 노드 함수 호출은 시간이 걸리기 때문에 이러한 낭비 요소를 반드시 고쳐주어야 합니다.

해결책은 시작할 때 한 번만 호출해서 그 결과를 변수에 저장해 두고 재사용하는 것입니다. 이벤트 비긴 플레이에서 'Get Owner'를 호출합니다. 리턴값을 드래그하여 변수로 승격(Promote to Variable)시킵니다. 변수 이름은 명확하게 'Parent Actor'로 지정합니다. 이제 틱 이벤트에서는 매번 함수를 호출할 필요 없이, 저장된 'Parent Actor' 변수를 연결해 주면 됩니다. 이렇게 수정하면 불필요한 반복 호출을 깔끔하게 방지할 수 있습니다.


## 엘리베이터 발판 실습
이제 본격적으로 발판을 만들어보겠습니다. 새로운 레벨을 생성합니다. 파일 메뉴에서 새 레벨, 베이직 레벨을 선택합니다. 이름은 '엘리베이터 맵'으로 저장해 줍니다. 실린더를 하나 배치하여 높은 기둥을 만듭니다. 높이와 너비를 충분히 키워줍니다. 그 앞에 큐브를 배치해 발판 모양을 만듭니다. 발판이 움직일 수 있도록 모빌리티는 '무버블(Movable)'로 설정합니다.

다음으로 이 발판에 위로 올라가는 기능을 추가해야 합니다. 이를 위해 블루프린트 클래스를 생성합니다. 원하는 기능이 있는 클래스가 없을 때는 기본적으로 '액터(Actor)'를 부모 클래스로 선택합니다. 하지만 3D 모델을 보여주는 기능이 이미 필요하므로, '스태틱 메시 액터(Static Mesh Actor)'를 상속받는 것이 조금 더 효율적입니다. 레벨에 배치된 발판을 선택합니다. 블루프린트 생성 버튼을 누릅니다. 이름을 'BP_FloatingElevator'로 지정하고 저장합니다.

이제 발판 위에 캐릭터가 올라온 것을 감지할 기능이 필요합니다. '박스 콜리전(Box Collision)' 컴포넌트를 추가합니다. 스태틱 메시와 겹치지 않게 박스를 살짝 위로 올려줍니다. 박스 콜리전을 선택하고 마우스 우클릭을 합니다. 'On Component Begin Overlap' 이벤트를 추가합니다. 이 겹침 이벤트에 프린트 스트링을 연결해 'On Board'라고 출력되게 합니다. 동일한 방식으로 'On Component End Overlap' 이벤트도 추가합니다. 여기에는 겹침이 끝났을 때 'Off Board'라고 출력되게 설정합니다. 실행해보면 캐릭터가 올라갈 때와 내릴 때 상황이 잘 감지되는 것을 볼 수 있습니다.

다음은 엘리베이터가 위로 올라가는 기능을 구현하겠습니다. 틱 이벤트에 'Add Actor Local Offset' 노드를 연결합니다. Z축에 1을 입력하면 매 틱마다 위로 올라가게 됩니다. 하지만 우리는 캐릭터가 밟았을 때만 올라가기를 원합니다. 여기에 브랜치(Branch) 노드, 즉 조건문을 추가합니다. 조건문에 연결할 변수를 생성하고 이름을 'OnBoard'로 지정합니다. 비긴 오버랩 이벤트에서 'OnBoard' 변수를 참(True)으로 설정합니다. 엔드 오버랩 이벤트에서는 거짓(False)으로 설정합니다. 이렇게 연결하면 캐릭터가 탑승했을 때만 엘리베이터가 위로 작동합니다.

마지막으로 출렁거리는 기능을 추가하겠습니다. 이전에 만들어 둔 '플로팅 컴포넌트'를 추가해 줍니다. 프리퀀시와 진폭 수치를 발판에 맞게 조금 낮춰줍니다. 하지만 엘리베이터가 올라가는 중에도 계속 출렁거리는 문제가 생깁니다. 탑승 시 출렁거림 기능을 끄고, 내릴 때 다시 켜주어야 합니다. 플로팅 컴포넌트 내부에서 'Turn On'과 'Turn Off'라는 함수를 만듭니다. 해당 함수 안에서 'Set Component Tick Enabled' 노드를 제어하여 틱 기능을 켜고 끕니다. 엘리베이터 블루프린트로 돌아와서, 탑승 시 'Turn Off'를, 내릴 때 'Turn On'을 호출하도록 연결합니다. 이때, 내릴 때의 위치에서 다시 출렁거려야 하므로 '시작 위치(Start Location)'를 현재 액터 위치로 다시 세팅해 주면 더욱 완벽합니다


# 물리 영역과 충돌 처리

물리 영역과 충돌 처리
이제 물리 영역 추가에 대해 알아보겠습니다. 레벨에 골드 트로피를 여러 개 배치해 봅니다. 테스트를 위해 트로피 근처에서 마우스 우클릭 후 '여기서 플레이'를 선택해 시간을 단축합니다. 현재 트로피를 먹으려고 부딪치면 캐릭터가 살짝 멈칫거리는 현상이 발생합니다. 이것은 물리적인 충돌을 강제하는 'Hit' 이벤트를 사용했기 때문입니다. 자연스럽게 아이템을 획득하려면 'Overlap' 이벤트를 사용해야 합니다. 트로피 블루프린트를 열어줍니다. 기존에 있던 히트 이벤트를 삭제합니다. 트로피를 감싸기 적합한 '캡슐 콜리전(Capsule Collision)' 컴포넌트를 추가합니다. 트로피 모양에 맞게 캡슐의 높이와 반경을 넓게 조정합니다. 캡슐 콜리전에 'Begin Overlap' 이벤트를 추가합니다. 이벤트 발생 시 아이템이 사라지도록 연결합니다. 다시 플레이해보면 캐릭터가 멈칫거림 없이 부드럽게 통과하며 아이템을 획득할 수 있습니다.
